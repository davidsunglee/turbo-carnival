## Bug Description

The player's ship can fly beyond **all four** screen boundaries (left, right, top, bottom) when moving at full speed into an edge. Happens on both iOS (touch) and macOS (keyboard). Additionally, after hitting a boundary, the ship gets "stuck" — reversing direction has no visible effect for an extended period.

## Root Cause

Two issues in the physics/clamp interaction:

1. **Clamp ordering:** `Galaxy1Scene.handleInput()` clamped `transform.position` **before** `PhysicsSystem.update()` ran. Physics then applied `position += velocity * dt`, overshooting the boundary every frame.

2. **Position desync:** `PhysicsSystem.syncFromComponents()` only synced **velocities**, not positions. So the clamp fixed `transform.position`, but the physics system's internal `positions[]` array kept drifting infinitely past the boundary. When the player reversed direction, the internal position had to walk back from (e.g.) x=-500 to x=-165 before any visual movement occurred.

## Fix

1. Moved the position clamp from `handleInput()` to immediately after `physicsSystem.update()` in `fixedUpdate()`.
2. Made `syncFromComponents()` also sync positions from `TransformComponent`, so external position changes (like the clamp) are respected by physics on the next frame.

## Files Changed

- `Engine2043/Sources/Engine2043/Scene/Galaxy1Scene.swift` — moved clamp block after physics update
- `Engine2043/Sources/Engine2043/ECS/Systems/PhysicsSystem.swift` — `syncFromComponents()` now also syncs positions
- `Engine2043/Tests/Engine2043Tests/Galaxy1SceneTests.swift` — added 5 tests (4 boundary + 1 immediate-response-after-reversal)
